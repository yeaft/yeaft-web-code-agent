01 · Motivation

Why build another code agent?为什么还要自己做 Code Agent？

Problem with black-box providers

Prompt hacks stack up.

Prompt hack 越叠越厚。

Claude Code is powerful, but its behavior is shaped by a large hidden prompt stack. Customizing on top of it often means fighting preloaded assumptions, RAI-related policy text, and provider-specific pacing. The result can be slower and harder to steer.

Claude Code 很强，但行为被大量隐藏 prompt 影响。基于它再改，常常是在和预置假设、RAI 相关文本、provider 自己的节奏对抗，结果就是速度慢、方向难控。

Yeaft's direction

Make the loop inspectable.

把执行循环变得可见。

Yeaft is built so the execution details are visible: what context was injected, which tool ran, how memory was recalled, how output was folded, and where model routing changed.

Yeaft 希望你能看到具体执行细节：注入了什么上下文、跑了哪个工具、记忆如何召回、长工具输出如何折叠、模型路由何时切换。

1. Less hidden prompt gravity

1. 减少隐藏 prompt 的重力

Start from a clean engine prompt, then add only the context the task needs.

从更干净的引擎 prompt 出发，只补任务真正需要的上下文。

2. More custom context

2. 更可定制的上下文

Project docs, memory, VP identity, runtime platform, tools, and debug traces are first-class inputs.

项目文档、记忆、VP 身份、运行平台、工具和调试 trace 都是一等输入。

3. Avoid passive compact

3. 避免被动 compact

Context compaction should be an explicit strategy, not a surprise loss of working state.

上下文压缩应该是显式策略，而不是突然丢失工作状态的被动事件。
