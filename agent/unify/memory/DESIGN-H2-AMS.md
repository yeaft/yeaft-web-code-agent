# DESIGN-H2-AMS — Active Memory Set + SQLite FTS Pre-flow

Supersedes prior router→worker memory-loading designs. The router
decision (which VP acts) is still needed for multi-VP fan-out, but
**memory loading is no longer LLM-gated on the hot path** — it's
pure-CPU FTS recall plus an **occasional** post-turn LLM adjustment.

This document is the spec for the H2.a–H2.f PR series.

---

## 0. Vocabulary

| Term | Definition | Storage |
|------|------------|---------|
| **Message** | Raw conversation message | `~/.yeaft/conversation/messages/*.md` |
| **Memory Segment** | A self-contained semantic chunk produced by Dream LLM from messages. May contain detail (multi-sentence, examples). | A region inside `~/.yeaft/memory/<scope>/memory.md` |
| **memory.md** | All segments for one scope, concatenated. Append-only from Dream; segments may be edited/merged in subsequent Dream passes. | Per scope file |
| **Summary** | Bounded compression of all segments in one scope. Regenerated by Dream after segment changes. | `~/.yeaft/memory/<scope>/summary.md` |
| **Scope** | `user` / `vp/<id>` / `group/<id>` / `feature/<id>` / `topic/<l1>[/<l2>]` | Directory path |
| **AMS** | Active Memory Set — the in-memory state object representing what's loaded for the current group session | RAM, per session |
| **Pre-flow** | Pre-turn pure-CPU pipeline: keyword extract → FTS → rerank → AMS update | engine pre-turn hook |
| **adjustMemory** | Post-turn LLM call that bidirectionally edits AMS membership | engine post-turn hook |

---

## 1. Memory Segment

A segment is **NOT** a copy of raw messages. Messages are already
persisted at `conversation/messages/*.md`; memory must be a higher-order
artifact, otherwise it's redundant.

A segment IS Dream LLM's secondary processing output: facts, decisions,
preferences, lessons, relations, goals, context — extracted and
structured. Detail is preserved (a segment can be many sentences and can
quote concrete examples). Segmentation is **semantic**: 30 turns of
chat with 3 distinct topics produces 3 segments, not one big blob.

### 1.1 Schema

```yaml
---
id: seg_<contentHash8>
scope: feature/auth
kind: decision        # fact | preference | decision | lesson | relation | goal | context
tags: [auth, jwt, session]
sourceMessages: [m_142, m_143, m_148]
createdAt: 2026-04-29T10:11:12Z
updatedAt: 2026-04-29T10:11:12Z
---
<segment body — natural language, multi-sentence, detail preserved>
```

### 1.2 Stability rules

- `id` is content-derived but stable across rewrites of unchanged content.
- Dream may **edit / merge / split** segments in later passes — when it
  does, it bumps `updatedAt` and **keeps the original `id` for the dominant
  successor**. AMS references survive the rewrite.
- adjustMemory **never** edits segment body — it only changes which
  segments are in the AMS.

### 1.3 Multi-scope segments

v1: one segment, one scope. If a memory is conceptually relevant to two
scopes, Dream emits two segments (with the same `sourceMessages`). v2
may add a `scopes: [...]` array; not in scope here.

---

## 2. memory.md ↔ segments

Physical file is multiple segments concatenated, separated by the YAML
frontmatter delimiter:

```
---
id: seg_a1b2
...
---
<body of segment 1>

---
id: seg_c3d4
...
---
<body of segment 2>
```

Parser `parseSegments(text) → Segment[]` splits on `---` boundaries
and validates frontmatter. Writer `serializeSegments(segs) → text`
preserves order. Round-trip must be lossless modulo whitespace.

---

## 3. Summary

`summary.md` is **derived from segments**, not from messages.

```
messages → [Dream extract] → segments → [Dream summarize] → summary.md
```

- One `summary.md` per scope.
- Hard cap: 800 tokens (configurable per scope-type; resident layer
  budget enforces this).
- Plain prose. No frontmatter (or minimal: `updatedAt` only).
- Regenerated whenever segments under that scope change.

Resident AMS layer = concatenation of all relevant scope summaries. It
is the high-precision, always-on context.

## 4. SQLite + FTS5 Index

Indexed unit = **segment** (not memory.md, not message).

### 4.1 Schema

```sql
CREATE TABLE memory_segments (
  id            TEXT PRIMARY KEY,
  scope         TEXT NOT NULL,
  kind          TEXT,
  tags          TEXT,           -- JSON array
  body          TEXT NOT NULL,
  source_msgs   TEXT,           -- JSON array
  created_at    INTEGER,
  updated_at    INTEGER
);
CREATE INDEX idx_segments_scope ON memory_segments(scope);
CREATE INDEX idx_segments_updated ON memory_segments(updated_at DESC);

CREATE VIRTUAL TABLE memory_fts USING fts5(
  body, tags, scope,
  content='memory_segments',
  content_rowid='rowid',
  tokenize='unicode61 remove_diacritics 2'
);

-- Triggers to keep FTS in sync
CREATE TRIGGER seg_ai AFTER INSERT ON memory_segments BEGIN
  INSERT INTO memory_fts(rowid, body, tags, scope)
  VALUES (new.rowid, new.body, new.tags, new.scope);
END;
CREATE TRIGGER seg_au AFTER UPDATE ON memory_segments BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, body, tags, scope)
  VALUES('delete', old.rowid, old.body, old.tags, old.scope);
  INSERT INTO memory_fts(rowid, body, tags, scope)
  VALUES (new.rowid, new.body, new.tags, new.scope);
END;
CREATE TRIGGER seg_ad AFTER DELETE ON memory_segments BEGIN
  INSERT INTO memory_fts(memory_fts, rowid, body, tags, scope)
  VALUES('delete', old.rowid, old.body, old.tags, old.scope);
END;
```

### 4.2 Disk → SQLite sync

Source of truth is the on-disk `memory.md` files. SQLite is a derived
index, rebuildable.

- On boot: scan `~/.yeaft/memory/**/memory.md`, diff vs. SQLite (by
  scope + segment id + updated_at), upsert / delete.
- After any Dream write: re-sync the affected scopes only.
- On schema mismatch / corruption: drop and rebuild.

## 5. Active Memory Set (AMS)

Per-group session state. In RAM, not persisted (rebuilt at session
start).

### 5.1 Layers

```
AMS = {
  resident:  Map<scope, summaryText>          // all relevant summaries
  recent:    LRU<segmentId, Segment>          // segments used in last N turns
  onDemand:  Set<segmentId>                   // loaded by FTS this turn
}
```

### 5.2 Budget

```
totalBudget = min(50_000, modelMaxContext * 0.10)

Layer split (initial — tunable):
  resident   40%   →  ~20k for a 200k model
  recent     25%   →  ~12.5k
  onDemand   35%   →  ~17.5k
```

If a layer exceeds its budget:
- **resident**: trim summaries by scope priority (current group's vp >
  user > feature > topic). Out-of-budget summaries are dropped from
  this turn's prompt only — the file isn't touched.
- **recent**: LRU evict.
- **onDemand**: keep top-K by FTS rank within budget.

### 5.3 Lifecycle

```
sessionStart:
  load summaries for: user, current group, all VPs in group, active
    features mentioned in group's last summary
  resident ← these summaries
  recent ← empty (or rehydrate top-K from last session — v2)
  onDemand ← empty

preTurn(userMsg):
  keywords = extractKeywords(userMsg)
  candidates = FTS query(keywords, scope-filtered)
  rerank by (scope match + tag overlap + recency)
  onDemand ← top-K within budget

turn:
  engine.query with full AMS injected

postTurn:
  recent.touch(used segments from this turn)
  if shouldRunAdjust: adjustMemory()

idle:
  schedule Dream
```

## 6. Pre-flow (no LLM)

```
preFlow(userMsg, ams, db):
  kws = extractKeywords(userMsg)            // existing keywords.js
  scopeFilter = ams.relevantScopes()         // group + VPs + active features
  rows = db.exec(`
    SELECT id, scope, kind, tags, body, snippet(memory_fts, 0, ...) as snip
    FROM memory_fts
    JOIN memory_segments USING (rowid)
    WHERE memory_fts MATCH :query
      AND scope IN (:scopeFilter)
    ORDER BY rank
    LIMIT 50
  `, { query: ftsQuery(kws), scopeFilter })

  reranked = rerank(rows, {
    scopeBoost: currentScope,
    tagOverlap: currentTags,
    recencyWeight: 0.2,
  })

  ams.onDemand = pickWithinBudget(reranked, onDemandBudget)
```

Latency target: < 10ms p95 on a 10k-segment index. No LLM.

## 7. adjustMemory (post-turn LLM, conditional)

### 7.1 Trigger

```
shouldRunAdjust =
     (newMemoryWritten   && ams.onDemand.size >= 5)
  || (turnTokenUsage      > totalBudget * 0.9)
  || (!session.adjustRanThisSession)    // first-turn guarantee
```

### 7.2 Inputs

LLM sees the **full visible memory** for the current group's reachable
scopes — there is no FTS-restricted candidate pool. "Visible" means
the same set the worker can see (private VP scopes filtered out).

```
adjustInput = {
  userMsg, assistantReply,
  currentAMS: { resident, recent, onDemand },
  visibleSegments: [
    { id, scope, kind, tags, body, inAMS: bool }
    for seg in db.scan(scopeFilter)
  ],
}
```

For very large scopes (>200 segments), pass segment summaries (first
sentence + tags) instead of full body, to keep prompt bounded.

### 7.3 Output

```yaml
add:    [seg_id, ...]   # not currently in AMS, should be
evict:  [seg_id, ...]   # currently in AMS, no longer needed
reason: "<one line>"
```

Apply to `ams.onDemand`. `add` may push past budget — in that case,
sort by adjustMemory's rank and trim, then log overflow.

### 7.4 What it does NOT do

- Does NOT edit segment body.
- Does NOT create new segments. (Dream does that.)
- Does NOT touch resident or recent layers directly (recent is LRU
  managed by usage; resident is summary-driven).

## 8. Dream (extract + summarize)

### 8.1 Per-scope prompts

Dream uses different extraction prompts per scope kind:

| Scope kind | Prompt focus |
|------------|--------------|
| `user`     | User identity, long-term preferences, habits, communication style |
| `vp/<id>`  | This VP's persona, role, interaction patterns with the user |
| `group/<id>` | The group's purpose, who's in it, current state |
| `feature/<id>` | Feature goal, decisions made, blockers, progress |
| `topic/<x>` | Core facts and viewpoints on this topic |

Each prompt template lives in `agent/unify/templates/dream-extract-<scope>.md`.

### 8.2 Extract pass

```
input:  recent messages since last Dream
output: segments[] with {scope, kind, tags, body, sourceMessages}
```

Dream LLM is told to produce **multiple segments per pass when topics
differ**, with explicit instructions to keep each segment self-contained.

### 8.3 Summarize pass

For each scope where segments changed:
```
input:  all segments in this scope
output: summary.md (≤800 tokens prose)
```

Old summary discarded; new summary written.

### 8.4 Maintenance pass (later — out of H2 scope)

Merge near-duplicate segments, retire stale ones. Uses an existing
Dream framework, just needs segment-aware adapters.

---

## 9. Privacy / ACL

- `vp/<other>` segments are filtered out of AMS for any worker that
  isn't `<other>` itself.
- adjustMemory's `visibleSegments` input applies the same filter.
- Pre-flow FTS query adds `scope NOT GLOB 'vp/<foreign>/*'` clauses.

---

## 10. Migration

R6 → v2 → segment-aware:

- The R6→v2 migration already exists (`migrate-r6-to-v2.js`).
- This design adds a v2 → segment migration:
  - Read each scope's `memory.md`.
  - If it's a single bulk doc (legacy v2 format), wrap as one segment.
  - Otherwise, parse existing segment delimiters.
  - Sync into SQLite.
- Run once on first boot after H2.a ships.

## 12. Out of scope (deferred)

- Multi-scope segments (one segment in two scopes simultaneously).
- AMS persistence across session restart (currently rebuilt fresh).
- Segment-level versioning / history (only latest body kept on disk).
- Cross-VP memory sharing protocols beyond the existing user-scope.
- Embedding-based recall (FTS + tags is the v1 retrieval model).
